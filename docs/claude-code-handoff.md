```
cd E:\PROJECTS\JuniorTorrentClient

Работай по задачам 1→7 по порядку; после каждой задачи — гейт build+test, бамп <Version> в JTC.csproj, коммит и git-тег vX.Y.Z-taskN по схеме версий, запись в docs/NIGHT_RUN_PROGRESS.md. > `--dangerously-skip-permissions` (YOLO-режим) даёт Claude Code выполнять правки, сборку и тесты без запроса подтверждений. Именно это нужно для сна. Все разрушительные действия ограничены правилами ниже.

### Железные правила ночного прогона

1. **Ветка, а не main.** Первым делом создать рабочую ветку: `git checkout -b night/stability-speed`. Ничего не коммитить в `main`/`master`.
2. **Никаких push и никаких сетевых релизов.** Только локальные коммиты и локальные теги. Не трогать `git remote`, не публиковать релизы, не запускать `tools/publish-github-release.ps1`.
3. **Каждый шаг = коммит + бамп версии + тег (чтобы можно было вернуться к любому шагу).** Порядок на каждой задаче:
   - гейт: `dotnet build src/JTC/JTC.csproj -c Debug` **и** `dotnet test tests/JTC.Tests/JTC.Tests.csproj` — оба зелёные;
   - поднять `<Version>` в `src/JTC/JTC.csproj` на +0.0.1 (см. схему версий ниже);
   - `git add -A && git commit -m "task N: <кратко> (vX.Y.Z)"`;
   - `git tag vX.Y.Z-taskN` — локальный тег на этом коммите.
   Один коммит + один тег = одна задача. Так к любому шагу можно вернуться через `git checkout vX.Y.Z-taskN`.
4. **Если гейт красный** — сделать до 2 попыток починить. Не вышло → `git stash` / откатить незакоммиченное по этой задаче (версию НЕ бампить, тег НЕ ставить), записать причину в прогресс-лог и **перейти к следующей задаче**. Не оставлять репозиторий в несобираемом состоянии.
5. **Только файлы из списка задач** плюс `<Version>` в `.csproj` и прогресс-лог. Не рефакторить смежное, не переформатировать чужой код, не трогать UI/XAML, не менять зависимости в `.csproj` (кроме поля `<Version>`).
6. **Никаких деструктивных операций с данными пользователя:** не удалять содержимое `%LocalAppData%\JTC`, не чистить `dist/`, не делать `git clean -x`, не удалять файлы вне репозитория.
7. **Прогресс-лог обязателен.** Вести `docs/NIGHT_RUN_PROGRESS.md`: по каждой задаче — статус (done / skipped / failed), суть изменений, хеш коммита, любые принятые допущения.
8. **Тупик — не блокировать всю ночь.** Любую задачу, требующую решения, которое нельзя принять безопасно, помечать `NEEDS HUMAN` в логе и идти дальше. Никаких ожиданий ввода.
9. **Границы времени/объёма.** Не уходить в бесконечные циклы правок одной задачи: правило «2 попытки — и дальше».

### Схема версий (текущая — 0.3.28)

Каждая успешно закоммиченная задача поднимает patch на единицу и получает тег:

| Задача | Версия | Тег |
|--------|--------|-----|
| 1 — per-torrent TorrentSettings | 0.3.29 | `v0.3.29-task1` |
| 2 — диагностическое логирование | 0.3.30 | `v0.3.30-task2` |
| 3 — watchdog авто-восстановления | 0.3.31 | `v0.3.31-task3` |
| 4 — защита сохранения состояния | 0.3.32 | `v0.3.32-task4` |
| 5 — кэширование настроек | 0.3.33 | `v0.3.33-task5` |
| 6 — периодический fast-resume | 0.3.34 | `v0.3.34-task6` |
| 7 — персист выбора файлов | 0.3.35 | `v0.3.35-task7` |

Если какая-то задача пропущена (красный гейт) — её номер версии не занимается, следующая задача берёт следующий свободный patch, а в прогресс-логе фиксируется, какой задаче какая версия соответствует. Откат к любому шагу: `git checkout vX.Y.Z-taskN` (или `git reset --hard` на нужный тег внутри ветки).

### Что можно делать полностью автономно, а что отложить до утра

- **Полностью автономно (код + сборка + юнит-тесты):** задачи **1, 3, 4, 5, 7** — у них есть проверяемый результат через `dotnet build`/`dotnet test`.
- **Реализовать код автономно, но помечать `NEEDS HUMAN VERIFY`:** задачи **2** (логирование — проверяется только живым прогоном закачки) и **6** (fast-resume — проверяется убийством процесса во время закачки). Код пишем ночью, ручную проверку оставляем на утро.
- **Не делать ночью:** ручные прогоны на реальных торрентах, замеры скорости «до/после», обновление релиз-ноутов и бамп `Version` — это финальные шаги для человека утром.

### Итог ночного прогона (сформировать в конце)

В конце Claude Code должен дописать в `docs/NIGHT_RUN_PROGRESS.md`:
- таблицу «задача → статус → коммит»;
- список `NEEDS HUMAN` пунктов;
- **утренний чек-лист** для тебя: что вручную проверить (см. «Финальная верификация»), как влить ветку (`git log night/stability-speed`, ревью diff, merge), что забампить перед релизом.

## Контекст проекта

Торрент-клиент на **WinUI 3 (Windows App SDK 2.2.0, unpackaged) / MonoTorrent 3.0.2 / .NET 10 / CommunityToolkit.Mvvm / xUnit**.
**Профиль нагрузки:** почти всегда одна активная закачка; две и более — редкость. Оптимизируем стабильность и скорость именно для одной горячей закачки.
**Симптомы, которые чиним:** обрывы/рестарты закачек и низкая скорость.

Ключевые файлы:
- `src/JTC/Services/TorrentService.cs` — обёртка над `ClientEngine`, добавление/удаление/очередь/восстановление.
- `src/JTC/Services/StateStore.cs` — персист списка торрентов (`torrents.json`).
- `src/JTC/Services/SettingsStore.cs` / `AppSettings.cs` — настройки приложения (`settings.json`).
- `src/JTC/Services/PersistedTorrent.cs` — запись о торренте для восстановления сессии.
- `src/JTC/ViewModels/MainViewModel.cs` / `TorrentViewModel.cs` — MVVM, таймер обновления UI 1 с.
- `tests/JTC.Tests/` — xUnit-тесты.

## Проверенные факты об API (уже сверено с `MonoTorrent.Client.dll` 3.0.2)

- У `ClientEngine.AddAsync` есть перегрузка, принимающая `TorrentSettings` (сейчас в коде используется вариант **без** настроек → торрент получает дефолтный per-torrent лимит соединений вместо глобальных 500).
- В сборке присутствуют: `TorrentSettingsBuilder`, `MaximumConnections`, `UploadSlots`, `AllowDht`, `AllowPeerExchange`, `MaximumDownloadRate`, `SaveFastResumeAsync`. Всё, что нужно ниже, реализуемо.

## Задача 1 (P0) — per-torrent `TorrentSettings` при добавлении  ⭐ главный прирост скорости

**Файл:** `src/JTC/Services/TorrentService.cs` (`AddTorrentFileAsync`, `AddMagnetAsync`).
**Проблема:** `_engine.AddAsync(torrentPath, downloadDir)` вызывается без `TorrentSettings`, поэтому каждый торрент упирается в дефолтный per-torrent лимит соединений (глобальные 500 при этом простаивают на единственной закачке).
**Сделать:**
1. Добавить константы рядом с существующими (`MaxPeerConnections`, `MaxHalfOpenConnections`):
   - `PerTorrentMaxConnections = 200`
   - `PerTorrentUploadSlots = 16`
2. Собрать настройки один раз (helper-метод `BuildTorrentSettings()`), используя `TorrentSettingsBuilder`:
   - `MaximumConnections = PerTorrentMaxConnections`
   - `UploadSlots = PerTorrentUploadSlots`
   - `AllowDht = true`, `AllowPeerExchange = true` (зафиксировать явно)
3. Передавать эти настройки в перегрузку `AddAsync(..., settings)` в обоих методах (файл и magnet).
4. Прокомментировать соотношение с глобальными лимитами (`MaximumConnections = 500`, `MaximumHalfOpenConnections = 50`) — почему 200 на торрент безопасно.

**Критерий приёмки:** сборка/тесты зелёные; в логе (после задачи 2) видно, что активная закачка набирает заметно больше подключённых пиров, чем раньше; глобальный лимит не превышается.

## Задача 2 (P0) — расширенное диагностическое логирование

**Файлы:** `src/JTC/Services/TorrentService.cs`, при необходимости `src/JTC/Services/DebugLog.cs`.
**Зачем:** без данных «обрывы» не диагностируются; нужен базовый замер до/после.
**Сделать:** периодический (раз в 5–10 с) снимок по каждой активной закачке в `debug.log`: состояние, скорость ↓/↑, число доступных и фактически подключённых пиров, seeds/leechers, статус трекеров, активность DHT. При переходе в `TorrentState.Error` — логировать `manager.Error?.Exception` (тип + сообщение + краткий stack).
**Критерий приёмки:** при обычной закачке в `debug.log` появляются периодические строки метрик; ошибка торрента фиксируется с причиной.

## Задача 3 (P0) — watchdog авто-восстановления после `Error`  ⭐ главный источник «обрывов»

**Файл:** `src/JTC/Services/TorrentService.cs` (`OnManagerStateChanged` и новый вспомогательный метод).
**Проблема:** упавшая в `TorrentState.Error` закачка висит мёртвой до ручного «Обновить».
**Сделать:** при переходе в `Error` запустить восстановление с ограничением попыток и нарастающей паузой (например, 5 с → 15 с → 60 с, максимум N=5 попыток на торрент, счётчик сбрасывается при успешном возобновлении скачивания). Стратегия: сначала `StartAsync`; если ошибки повторяются — один `HashCheckAsync` + старт. Уважать фатальные причины (нет места на диске, нет прав) — на них не зацикливаться, только залогировать и остановиться.
**Критерий приёмки:** намеренно вызванная временная ошибка (например, кратковременная недоступность каталога) авто-восстанавливается без действий пользователя; на фатальной ошибке ретраи прекращаются после лимита. Добавить юнит-тест на политику ретраев (лимит + backoff), выделив её в тестируемый метод.

## Задача 4 (P0) — обезопасить сохранение состояния от гонок

**Файлы:** `src/JTC/Services/StateStore.cs`, `src/JTC/Services/SettingsStore.cs`, `src/JTC/Services/TorrentService.cs`.
**Проблема:** `SaveAsync` использует общий `*.tmp`-файл и не сериализован; `OnManagerStateChanged` — `async void` с `catch {}`, вызовы `SaveStateAsync` могут идти конкурентно → редкие `IOException`, риск порчи `torrents.json`.
**Сделать:**
1. В `StateStore` (и по аналогии в `SettingsStore`) — уникальное имя временного файла (`Path.GetRandomFileName()`), затем атомарный `File.Move(..., overwrite: true)`.
2. Ввести `SemaphoreSlim`, сериализующий запись состояния.
3. В `OnManagerStateChanged` не глотать исключения молча — логировать через `DebugLog.Error`.
**Критерий приёмки:** новый юнит-тест: параллельные вызовы сохранения не бросают и оставляют валидный JSON. Существующие `StateStoreTests` проходят.

## Задача 5 (P1) — кэшировать настройки вместо чтения с диска в цикле

**Файл:** `src/JTC/Services/TorrentService.cs` (`CanStartMore`, `StartNextIfSlotFreeAsync`, `ApplySettingsAsync`).
**Проблема:** `CanStartMore()` дергает `SettingsStore.Load()` (синхронное чтение файла) внутри `while` и на каждом изменении состояния.
**Сделать:** хранить актуальные `AppSettings` в поле, обновлять в `ApplySettingsAsync`, читать из памяти.
**Критерий приёмки:** дисковых чтений настроек в горячем пути нет; логика очереди не изменилась (тесты очереди зелёные).

## Задача 6 (P1) — периодическое сохранение fast-resume

**Файл:** `src/JTC/Services/TorrentService.cs`.
**Проблема:** `AutoSaveLoadFastResume = true` сохраняет только при штатной остановке → после краша/kill следующий старт делает полный рехеш (долгий старт, нулевая скорость).
**Сделать:** таймер раз в 30–60 с, сохраняющий fast-resume активных закачек (`TorrentManager.SaveFastResumeAsync` или API движка). Учесть I/O-нагрузку на HDD/SMR — не чаще 30 с.
**Критерий приёмки:** после симулированного краша (убить процесс во время закачки) следующий запуск не выполняет полный рехеш, а поднимает прогресс из fast-resume.

## Задача 7 (P1) — персист выбора «не качать эти файлы»

**Файлы:** `src/JTC/Services/PersistedTorrent.cs`, `src/JTC/Services/StateStore.cs`, `src/JTC/Services/TorrentService.cs` (`AddTorrentFileAsync`, `LoadStateAsync`).
**Проблема:** `skipFileIndices` (файлы с `Priority.DoNotDownload`) не сохраняются → после перезапуска исключённые файлы качаются заново.
**Сделать:** добавить в `PersistedTorrent` массив индексов пропущенных файлов (обратносовместимо: старые записи без поля), сохранять при добавлении и повторно применять в `LoadStateAsync`.
**Критерий приёмки:** юнит-тест round-trip персиста со `skipFileIndices`; после перезапуска исключённые файлы остаются `DoNotDownload`.

## Риски

- **Число соединений (задача 1):** слишком высокие per-torrent значения упираются в лимиты Windows на полуоткрытые TCP и дают всплески CPU/памяти. 200 — безопасный старт; выше — только по данным логов.
- **Авто-ретрай (задача 3):** обязательно ограничить попытки и уважать фатальные ошибки, иначе бесконечный цикл «старт → ошибка → старт».
- **fast-resume таймер (задача 6):** частая запись на HDD/SMR = I/O-всплески; 30–60 с — компромисс.
